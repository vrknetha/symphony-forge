## Problem Statement

CAW Studios wants one harness for every new client repo, covering the full lifecycle: discovery → prototype → client sign-off → nx-workspace scaffold → per-feature plan/implement/test/review/PR — with every product and engineering decision captured in the repo ("self-evolving repository"). Symphony Forge today covers only the post-docs factory half, is Codex-only, and doesn't integrate the org constitution (`kl-engg-constitution`) or the chosen skill packs (gstack, mattpocock/skills, ponytail, openclaw autoreview). Claude Code must be the enforced coordinator (planning, ideation, orchestration via openai/codex-plugin-cc); Codex is the execution/review plane. Hard requirement: the harness ships both `.claude/` and `.codex/` directories, each holding runtime-specific assets, referencing shared canonical files — never duplicating them.

## Demand Evidence

- Leadership mandate: the harness will be organizationally required for new CAW repos (D2).
- Acknowledged risk (agreed in session): mandate produces compliance, not usage — devs run required commands while thinking outside the system. The design therefore optimizes the dev's daily loop, not just the gates.
- Three named status-quo pains (D3): **lost decisions** (Slack/calls/heads; reconstruction expensive when devs rotate or clients ask "why"), **inconsistent AI usage** (every dev prompts differently; quality unreviewable; good patterns don't spread), **slow project starts** (every repo set up differently; standards drift from playbooks).
- The maintainer has already run 3 wipe-rebuild experiment rounds through the factory (git history R1–R3) — the pain is proven on his own work; no other CAW dev has run it yet.

## Status Quo

New CAW projects start ad-hoc: repos set up differently each time, playbooks live in Notion (synced to the `kl-engg-constitution` skill), each dev drives Claude/Codex/Cursor their own way, and decisions live in Slack threads and memories. Symphony Forge exists as a Codex-only factory template with deterministic phase gates (`.codex/scripts/*.py`, `.factory/*.json`), 7 Codex subagents, and Linear-first task state — but no discovery/prototype phase, no Claude Code entry point, and no constitution integration.

## Target User & Narrowest Wedge

- **Primary feeler of the pain:** the platform owner (maintainer of both symphony-forge and kl-engg-constitution), accountable for standards drift and lost decisions across projects. **Daily user:** the delivery dev. Named design rule (agreed): decision capture must be *exhaust* from the dev's normal workflow — from question answers, plan approvals, and sign-offs — never a form.
- **Narrowest wedge (validated independently by the Codex cold read):** repo bootstrap + decision capture + sign-off gate. `forge init` produces a new client repo with dual-runtime layout, shared canonical docs, discovery templates, and a gate that blocks heavy factory phases until sign-off is recorded (mechanism below).

**Phase vocabulary (used consistently below):** phase **0a = discovery** (problem, stakeholders, business decisions), phase **0b = prototype** (throwaway build for client sign-off). Both precede the existing factory phases (`planning` onward) and run lightweight — no `.factory` ceremony.

**Decision-record schema (the contract the linter and gates read):** `docs/decisions/NNNN-slug.md` with YAML frontmatter: `status: proposed|accepted|superseded`, `confirmed_by: <name>` (must be non-empty for `accepted`), `date`, optional `supersedes`. The commit adding an `accepted` record must carry a `Confirmed-by:` git trailer matching a human author. This is a **convention-level control, checked structurally** — the linter verifies field presence and the trailer, it cannot verify a human actually read it; that residual trust is accepted and documented.

**Sign-off gate mechanism:** `.agents/scripts/record_signoff.py` verifies a decision record with slug `client-signoff` (any `NNNN-client-signoff.md`) exists with `status: accepted` and non-empty `confirmed_by`, then sets `client_signoff: true` in `.factory/run.json`. This is **new gating code**: extend `update_run.py` and the pre-tool hook to refuse any phase ≥ `planning` while `client_signoff` is absent or false (no such precondition exists in today's factory). Record numbering: a helper (`forge decision new <slug>`) assigns the next free integer; the linter flags duplicate numbers. Note on Premise 5: a minimal `.factory/run.json` (project identity + signoff flag) written during 0a/0b is not "ceremony" — decomposition, tests, and review artifacts remain post-sign-off only.

## Constraints

- Dual-runtime: `.claude/` and `.codex/` both first-class; shared canonical files (AGENTS.md, `docs/`, conventions) referenced from both sides. Enforced by a linter, not convention.
- **Structural split (D8, user decision, held against cross-model advice):** Claude Code is THE coordinator; Codex is the execution/review plane via codex-plugin-cc. Required mitigations: delegation must be fast (`/codex:rescue --background`, no copy-paste handoffs) and a documented degraded mode must exist for when codex-plugin-cc is unavailable (sanctioned fallback, not improvisation).
- codex-plugin-cc facts (verified 2026-07-09): slash commands + `codex-rescue` subagent + hooks wrapping the global `codex` CLI — not MCP; Claude-side CLAUDE.md does **not** propagate to Codex subprocesses (shared files must live where Codex reads: AGENTS.md, `.codex/config.toml`); the optional Stop-hook review gate causes open-ended review loops — keep it **disabled**; explicit `/codex:review` at the review phase instead.
- **Constitution merged into symphony-forge (D10/D12 revision):** the org playbooks move from the separate `kl-engg-constitution` skill into `constitution/` inside symphony-forge, which becomes **the** single source of truth — Notion is a one-time import source and is then deprecated (no ongoing sync). Standards changes land exclusively as PRs to symphony-forge, giving every rule change a diff, a review, and a rationale — the constitution governed by the same decision-capture discipline it mandates. Structurally better for dual-runtime too: standards are repo files readable by both Claude Code AND Codex (Codex subprocesses can never see `~/.claude` skills). The old precedence rule (playbooks > gstack > other third-party skills) carries over into `harness.yaml`. (A human-readable Notion mirror *generated from* the repo can be added later if leadership wants one — explicitly deferred, never authoritative.)
- nx-workspace is the preferred scaffold target where possible (constitution's monorepo standard should name it explicitly — see Dependencies).
- Linear remains task-state source of truth; the repo owns decisions.
- Skill-pack reality (researched this session): gstack and mattpocock/skills are competing full pipelines; ponytail is a code-minimalism bias layer that will attack CAW's NestJS/FastAPI boilerplate and `{success,data,error}` envelope; openclaw autoreview is the one composable part (deterministic `codex exec` review loop, JSON verdicts, "never nest reviewers"). Uncurated LLM-generated context measurably hurts (−3% success, +20% cost).

## Premises

1. **Adoption beats mandate** — the dev's daily loop must be faster with the harness than ad-hoc, or the mandate produces compliance theater. Decision capture is exhaust, never a form.
2. **One owner per lifecycle phase** — discovery (0a), prototype (0b), planning, implementation, review, ship each get exactly one owning pipeline; overlapping skills are scoped or disabled. Decided in v1: codex-plugin-cc's Stop-hook gate disabled; gstack's `/codex` and `/ship` disabled inside factory repos; mattpocock pack not installed; **ponytail allowed only in phase 0b (prototype), lite mode** — excluded from all factory phases. These are decisions, not open questions; the pilot may propose reversals via the normal evolution loop.
3. **Repo is the decision store** — `docs/decisions/` + `docs/product/BRIEF.md` in the generated repo; not `~/.gstack` (machine-local), not Linear-first for decisions. Agent-drafted decision records are human-confirmed, because agents fabricate the *why*.
4. **Constitution rules, referenced everywhere (revised at D10/D12)** — the constitution lives IN symphony-forge (`constitution/`) as the single source of truth (Notion deprecated after one-time import; all changes via PR); `harness/*/conventions/` and runtime configs reference it, not restate it. `forge init` vendors it into generated repos (pinned version) so every project carries its standards in-repo where both runtimes read them. The separate kl-engg-constitution skill is retired or reduced to a thin shim for non-forge repos (open question).
5. **Heavy gates only after sign-off** — discovery/prototype runs lightweight (no `.factory` ceremony); the full factory applies per-feature after client sign-off. (Full SDD ceremony on everything is reinvented waterfall — documented ~10x overhead failures.)
6. **Self-evolution = curated evolution** — learnings flow through the platform owner (retro → proposed constitution/harness PR → human merge); agents never auto-append docs or skills.

(All six accepted at D6. Session eureka: the harness's real job is **precedence and subtraction, not skill accumulation** — a thin curation kernel assigning each phase one owner.)

## Approaches Considered

### Approach A: The Bootstrap Wedge (minimal viable)
`forge init` (nx-workspace + dual-runtime layout + shared core) + discovery→sign-off gate + dual-runtime linter; existing factory unchanged post-sign-off; codex-plugin-cc wired with Stop-hook disabled. Effort M / Risk Low. Ships in days; smallest diff; Codex independently converged on this scope. But leaves skill-pack collisions unresolved (pain #2 persists) and decision capture stays manual.

### Approach B: The Two-Plane Factory (ideal architecture)
Full v1: **Studio plane** (pre-sign-off, lightweight — Claude Code + curated gstack subset writing exhaust into BRIEF + decisions) and **Factory plane** (post-sign-off, enforced Claude→Codex via codex-plugin-cc, openclaw autoreview as the single review gate), plus a **precedence manifest** (`harness.yaml`) declaring one owner per phase and generating both runtimes' config from the constitution, plus a curated evolution loop (retro → constitution PR). Effort XL / Risk High. Delivers the whole vision by construction, but is a big-bang platform build — the documented failure mode for mandated IDPs — with weeks of building before any dev feedback.

### Approach C: Extract the Harness from a Pilot (creative/lateral) — CHOSEN
Build A's wedge in week one, then run **one real CAW client project** end-to-end (discovery → sign-off → nx scaffold → enforced Claude→Codex features) with the platform owner + one named champion dev. Every friction point becomes a harness commit or a constitution PR that week. B's pieces (precedence manifest, decision-exhaust automation) are built only when and in the order the pilot demands. After one project, promote what survived into symphony-forge v2; roll out org-wide with the pilot as reference implementation and the champion as internal advocate.

## Recommended Approach

**Approach C** (selected at D9). Rationale: demand evidence is a mandate, not pull; champion-led pilots are the documented fix for mandate-driven platform death; C contains A immediately and earns B with evidence instead of speculation.

Week-one target (A-scope, inside symphony-forge — ambitious for the window; slipping an item is acceptable, reordering is not):
1. **`.agents/`-centered layout (D14 revision)**: `.agents/` is the universal home for all runtime-neutral agent substance — phase prompts (from `.codex/prompts/`), subagent role definitions (neutralized from `.codex/agents/*.toml`), hook logic scripts, and the deterministic factory scripts (from `.codex/scripts/`). `.codex/` and `.claude/` shrink to **thin registration adapters**: `.codex/config.toml` + `hooks.json` + agent TOMLs that only point into `.agents/`; `.claude/` settings + CLAUDE.md pointer enabling codex-plugin-cc (review-gate off). Shared canon: `AGENTS.md`, `docs/**`, `harness/**`, `constitution/**`, `.agents/**` — referenced from both runtimes, duplicated in neither. (Neither runtime discovers `.agents/` natively — the adapters are mandatory plumbing, and the linter enforces they stay thin.)
2. **`forge` CLI**: a single typer app at `.agents/scripts/forge.py` with subcommands `forge init` and `forge decision new <slug>` (all deterministic scripts live in `.agents/scripts/` per the D14 layout). Scaffolds a new client repo — nx-workspace target, dual-runtime dirs, BRIEF/architecture/decisions templates (schema above), phase-0a discovery templates, sign-off gate wiring. **Init contract: fails if the target directory is non-empty; `--force` required to overwrite; no merge mode in v1.**
3. **Dual-runtime linter** at `.agents/scripts/check_dual_runtime.py` — all deterministic scripts (including the existing `.codex/scripts/*.py`) migrate to `.agents/scripts/` as part of the D14 layout; the `.codex/hooks.json` adapter re-points its commands there. All checks are structural — no semantic judgment: (a) shared canon files exist; (b) **forbidden-filename rule**: no file under `.claude/` or `.codex/` may share a basename with any canonical convention/constitution file; (c) **duplication rule**: no file under runtime dirs is a byte-copy of a canonical file, or contains **≥10 consecutive identical non-blank lines** from one; (d) runtime files referencing standards must use path references into `docs/`/`harness/`/AGENTS.md (marker: a `<!-- canon: <path> -->` line); (e) every `accepted` decision record has non-empty `confirmed_by` + `Confirmed-by:` git trailer, and record numbers are unique; (f) `.claude/` and `.codex/` configs resolve to the same canonical paths; (g) **thin-adapter rule (D14)**: `.claude/` and `.codex/` may contain only registration/config files (config.toml, hooks.json, settings, pointer files) — any prompt text, role definition, or script logic found there must live in `.agents/` instead.
4. **Skill curation v0** (precedence manifest): **`harness.yaml` at repo root** (machine-readable from day one; the linter reads it, `docs/SKILLS.md` can be generated from it later). Declares per-phase owners — 0a discovery: gstack `/office-hours` + design skills; 0b prototype: `prototype` skill + ponytail-lite; planning: Claude Code plan mode (+ constitution) — **with codebase exploration delegated to Codex (D16)**: the Claude planner does not grep/read the codebase itself; it dispatches read-only Codex investigations (`/codex:rescue --background` or `codex exec -s read-only` with the exploration question) and plans from the returned findings. Rationale: extends the D8 split into planning (Codex traces code paths more carefully; Claude's context stays lean for synthesis), and exploration findings arrive as citable reports that can feed decision records. Degraded mode applies here too; implementation: `/codex:rescue` (factory prompts); review: **owner = the three existing subagents** (RESOLVED at D15 — they are standards-aware and already emit the three hard-gate artifacts), upgraded with autoreview's loop discipline in their prompts (scope-freeze baseline, verify findings against real code paths, stop after 2 non-converging cycles). **openclaw autoreview is a named escalation tier, not a per-task gate**: run at PR-ready only for tasks flagged security-sensitive / migration / cross-domain refactor, where an independent standards-blind adversary earns its cost (optionally as a `--reviewers codex,claude` panel); its findings feed the subagent artifacts via a thin adapter. Never both loops on the same task ("never nest reviewers"). Ship: factory `pr_ready.py`.
5. **Degraded mode doc**: what a dev does when codex-plugin-cc breaks (direct `codex exec` with the same phase prompts; artifacts identical).
5b. **Hook de-noising (D14)** — principle: *quiet by default, loud only at phase gates*. Context injection at most once per session; hooks are silent unless blocking; blocking only on phase-transition violations, never keywords/exit-codes/turn-ends. Concretely:
   - `session_start.py`: keep, but dynamic run-state lines only (drop the 4 static lines restating AGENTS.md).
   - `user_prompt_submit.py`: **delete** the keyword sniffing ("implement/build/fix/ship" blocking has false positives both ways) and the every-prompt "if vague…" lecture; phase enforcement moves to the pre-tool gate + `update_run.py`.
   - `post_tool_use.py`: **delete** the per-command context injection and the block-on-any-non-zero-exit rule (grep/probe/expected-failure commands legitimately exit non-zero); tool history trimmed to command + exit code, or dropped (the transcript already has it).
   - `stop_continue.py`: **delete** the artifact-hostage behavior (it duplicates `pr_ready.py`'s deterministic gate and, combined with codex-plugin-cc's Stop gate, is the documented loop recipe); at most a one-line non-blocking warning when phase == implementing with uncommitted changes.
   - `pre_tool_use.py`: keep as the single runtime gate (command policy + `client_signoff` phase check).
6. **Harness self-test**: a CI smoke test that runs `forge init` into a temp dir, then runs the dual-runtime linter and the scaffold-structure check (`check_factory_scaffold.py`) against the generated repo — **not** the full factory `verify.py`, which expects decomposition/tests/review artifacts that a fresh scaffold legitimately lacks. Wired into `.github/workflows/factory-scaffold.yml`.

Pilot phase (weeks 2+): run the client project; each friction point → harness/constitution PR; decision-exhaust automation (drafting ADRs from AskUserQuestion answers and sign-offs) gets built when the pilot's first manual ADR feels like a form.
